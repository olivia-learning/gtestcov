# gtestcov Project Structure

This document explains what each top-level directory is for. It also separates
the `gtestcov` tool repository from the C++ project where `gtestcov` is used.

## Tool Repository

The tool repository is the checkout of `olivia-learning/gtestcov`.

```text
gtestcov/
  src/gtestcov/       # Python package and CLI/MCP implementation
  tests/              # gtestcov's own Python test suite
  examples/           # small public example projects used by tests and demos
  docs/               # user-facing docs, diagrams, and install guide
  pyproject.toml      # Python package metadata, dependencies, and test config
  README.md           # project overview and document navigation
  AGENTS.md           # rules for AI coding agents working in this repo
```

### `src/gtestcov/`

This is the tool implementation. Important groups:

- CLI and integration: `cli.py`, `mcp_server.py`, `opencode.py`
- Project understanding and evidence: `codrax.py`, `understanding.py`,
  `discovery.py`, `profile_sync.py`, `file_index.py`,
  `evidence_backend.py`, `evidence_pack.py`, `search_backend.py`,
  `semantic_backend.py`
- Test planning: `analyzer.py`, `obligations.py`, `task.py`, `cover.py`
- Safety and verification: `audit.py`, `permissions.py`, `preflight.py`,
  `verify.py`, `diagnose.py`, `next_round.py`
- Context and lifecycle: `memory.py`, `upgrade.py`, `version.py`
- Shared models and helpers: `models.py`, `profile.py`, `fs.py`,
  `coverage_goal.py`, `dependency.py`, `adapters.py`

Project-specific facts must not be hardcoded here unless they are gated by
CODRAX file:line evidence, user input, profile data, or tool artifacts. Local
file indexes and bulk symbol scans provide deterministic candidate evidence;
Zoekt and semantic backends such as Serena, clangd, and ccls remain optional
PoC/fallback paths, not default hard dependencies.
`evidence_pack.py` stores v3-only evidence pack cache data with
`payloads.codrax`; older cache layouts are treated as cache misses.

### `tests/`

This is the regression suite for the tool itself. It validates CLI behavior,
CODRAX evidence handling, profile sync, task generation, preflight checks,
verification, memory, upgrade/rollback, and public example workflows.

### `examples/`

This contains small public projects used as safe test fixtures. These are not
the user's real embedded project. Do not infer real project structure from the
example names.

### `docs/`

This is user-facing documentation.

- `install.md`: the single source of truth for install, AI-assisted install,
  upgrade, restore-custom, and rollback.
- `gtestcov_workflow.md`: readable workflow diagram in Markdown.
- `gtestcov_workflow.drawio`: editable diagrams.net / draw.io workflow.
- `project_structure.md`: this file.

### Release Artifacts

`Out_of_the_box_ready/` contains ready-to-use release assets. The checked-in
`Out_of_the_box_ready/gtestcov-v0.4.0.zip` is the current local
release-preparation package. `Out_of_the_box_ready/gtestcov-v0.3.0.zip` is kept
only as a historical release artifact. Future release packages should be rebuilt
from final code and docs, include a matching `gtestcov_install_manifest.json`,
and use a zip filename that matches the package version.

## User C++ Project

The user C++ project is selected by `--project-root`. It does not have to be a
`.git` repository root. It may be a source archive, a `.repo` workspace, a
monorepo subdirectory, or another user-selected workspace.

`gtestcov` writes project-local state under that root:

```text
<project-root>/
  project_profile.yaml       # project profile generated or updated by gtestcov
  .gtestcov/                 # gtestcov run state, reports, memory, and upgrade snapshots
  .opencode/commands/        # OpenCode command files generated by gtestcov init
```

### `project_profile.yaml`

This records project-specific build/test/coverage settings and evidence-backed
test support paths. Users should not have to write it from scratch; `gtestcov`
and CODRAX evidence should fill it in where possible.

CODRAX long-running requests are controlled by `idle_timeout_seconds`,
`max_runtime_seconds`, `probe_timeout_seconds`,
`status_update_interval_seconds`, `native_log_tail_bytes`, and
`final_log_max_bytes`. There is no separate current `timeout_seconds` field for
the main CODRAX request.

### `.gtestcov/`

This is generated project state. Typical contents:

```text
.gtestcov/
  runs/<run_id>/             # decision reports, task packages, checks, verify results
  memory/                    # project memory and handoff material
  profile_backups/           # profile backups before profile-sync changes
  upgrade_slots/<id>/        # old/new project snapshots for upgrade and rollback
  upgrade_state.json         # active upgrade metadata
```

Weak AI may read `.gtestcov/` evidence and task files, but must not directly edit
tool-generated state such as handoff files, project memory, verify results, or
coverage history.

Inside a run directory, `gtestcov_status.json` records the current gtestcov
phase and `codrax_status.json` records the current CODRAX phase. During CODRAX
execution, native CODRAX logs are kept under `codrax_native_logs/`. After each
CODRAX invocation exits, `codrax_final_outputs/` stores a bounded final-output
record for that invocation, and `codrax_final_log.md` points to the latest
invocation content. The structured `codrax_evidence.*` files are still the
source of project facts.

### `.opencode/commands/`

`gtestcov init` writes OpenCode command files here. These command files tell
OpenCode how to call `gtestcov` tools and where to find the single install guide
or coverage workflow instructions.

## What Not To Mix Up

- `src/gtestcov/` is the tool code. It should stay generic.
- `<project-root>/project_profile.yaml` is project-specific evidence and config.
- `<project-root>/.gtestcov/` is generated run state for one C++ project.
- `examples/` is only a public fixture area, not a template for the user's real
  project.
- `.pytest_cache/`, `__pycache__/`, build outputs, and coverage outputs are
  generated artifacts and should not be treated as source documentation.
