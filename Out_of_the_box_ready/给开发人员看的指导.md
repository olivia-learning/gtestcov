# 给开发人员看的指导

这份文档只回答一个问题：

```text
开发人员应该怎么对 OpenCode + 弱 AI 说，才能让它严格按照 gtestcov 工具走，而不是中途按自己的思路跑偏。
```

## 一句话原则

不要说：

```text
帮我给这个文件写单测。
```

要说：

```text
请严格执行 gtestcov 工作流。你不能自由写测试，只能按 gtestcov 生成的 task package、证据和检查结果行动。
```

## 1. 开始一个覆盖率任务时，直接这样说

把下面这段复制给 OpenCode / 弱 AI：

```text
请严格执行 /gtest-cover，不要自由发挥。

目标文件：<填写目标文件路径>
目标行覆盖率：<填写目标覆盖率，例如 80%>
编译入口：<填写编译入口文件，可以是 CMakeLists.txt、.sh、Makefile、BUILD.gn 或项目真实构建入口>

要求：
1. 先调用 gtestcov 工具生成 task package。
2. 写测试前必须阅读 gtestcov 生成的 resume_prompt.md、handoff.md、project_memory.md、task.md 和 opencode_permission_warmup.md。
3. 项目事实只能来自用户输入、project_profile.yaml、CODRAX file:line 证据或 gtestcov 产物。
4. 只能修改 task.md 允许的测试、测试 support、测试构建配置路径。
5. 禁止修改目标文件和业务逻辑源码。
6. 证据不足时写 manual_review_needed.md，不要猜。
7. 需要业务代码改动时写 source_change_request.md，不要直接改。
8. 写完测试后必须先运行 gtestcov check，通过后才运行 gtestcov verify。
9. 覆盖率不达标时读取 next_task.md，再进入下一轮。
10. 如果出现 stagnation_report.md，立即停止并让我 review。
```

## 2. 如果弱 AI 开始自己分析项目，立刻这样打断

```text
停。不要自由探索项目。

请回到 gtestcov 工作流：
1. 先读当前 run 的 task.md。
2. 只使用 gtestcov 整理出的 CODRAX file:line evidence。
3. 不要根据文件名、目录名或经验猜项目结构。
4. 如果证据不够，写 manual_review_needed.md。
```

## 3. 如果弱 AI 想改业务代码，立刻这样说

```text
停。禁止直接修改业务逻辑源码或目标文件。

如果你认为必须改业务代码，请只写：
.gtestcov/runs/<run_id>/source_change_request.md

然后停止，等待我 review。
```

## 4. 如果弱 AI 想跳过 check 直接编译，立刻这样说

```text
停。不能直接编译。

必须先运行：
gtestcov check --project-root . --run-id latest --target <目标文件>

只有 check 通过后，才允许运行：
gtestcov verify --project-root . --run-id latest
```

## 5. 如果弱 AI 上下文刷新后像失忆了，直接这样说

```text
你现在必须先恢复上下文，不要继续写代码。

请先阅读：
1. .gtestcov/runs/<run_id>/resume_prompt.md
2. .gtestcov/runs/<run_id>/handoff.md
3. .gtestcov/memory/project_memory.md
4. .gtestcov/runs/<run_id>/task.md

读完后，只按这些文件里的下一步继续。
```

## 6. 如果覆盖率没达标，直接这样说

```text
不要自己猜下一轮怎么补。

请读取：
.gtestcov/runs/<run_id>/next_task.md

然后只按照 next_task.md 规划的方向补测试。
如果 next_task.md 不存在，请先运行 gtestcov verify 或 gtestcov next-round。
```

## 7. 如果构建、测试或覆盖率失败，直接这样说

```text
不要自己猜失败原因。

请运行：
gtestcov diagnose-failure --project-root . --run-id latest --target <目标文件>

然后只修测试代码、测试 support 或测试构建配置。
禁止修改业务逻辑源码。
```

## 8. 批准文件权限时，按这个原则

可以批准：

```text
读取目标文件
读取 gtestcov 证据文件
读取构建入口
读取已有测试
修改 task.md 允许的测试文件
修改 task.md 允许的测试 support
修改 task.md 允许的测试构建配置
```

不要批准：

```text
修改目标文件
修改业务逻辑源码
修改 handoff.*
修改 project_memory.*
修改 verify.json
修改 coverage_history.json
修改 gtestcov 工具生成状态文件
```

## 9. 最短可复制版本

如果你只想贴一段，就贴这个：

```text
请严格按照 gtestcov 工作流执行，不允许自由改流程。

目标文件：<target-file>
目标行覆盖率：<percent>
编译入口：<build-entry-file>

先使用 /gtest-cover 或 gtestcov MCP 工具生成 task package。
开始写代码前必须阅读 resume_prompt.md、handoff.md、project_memory.md、task.md、opencode_permission_warmup.md。
只能修改 task.md 允许的测试、测试 support、测试构建配置路径。
禁止修改目标文件和业务逻辑源码。
项目事实必须引用 gtestcov 整理的 CODRAX file:line evidence。
证据不足时写 manual_review_needed.md，不要猜。
需要业务代码改动时写 source_change_request.md，不要直接改。
写完测试后先运行 gtestcov check，通过后才运行 gtestcov verify。
覆盖率不达标时读取 next_task.md，再进入下一轮。
如果出现 stagnation_report.md，停止并让我 review。
```

## 10. 判断弱 AI 是否跑偏

只要它出现下面任何行为，就说明可能跑偏：

```text
没读 task.md 就开始写测试
没引用 CODRAX file:line evidence 就判断项目结构
想直接改业务源码
想跳过 gtestcov check
想直接运行全量编译但没有说明为什么
覆盖率不够时自己编下一轮策略
上下文刷新后不读 resume_prompt.md / handoff.md
```

这时不要继续让它写，直接用上面的打断话术把它拉回 gtestcov。
